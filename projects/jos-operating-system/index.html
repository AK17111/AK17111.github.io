JOS — Unix-like Operating System

Building an x86 Unix-like OS from scratch: boot loader, virtual memory, user environments, preemptive multitasking, and a file system with a shell
Purdue University · Operating Systems Spring 2026 C · x86 Assembly · QEMU
JOS is the operating system built across the lab sequence from MIT's 6.828: Operating System Engineering, which Purdue's OS course uses as its lab curriculum. I worked through Labs 1–5 with my lab partner, Anas A. (fellow Purdue student), pair-programming the whole way rather than splitting labs between us. Since these are live, reused course assignments, the repo stays private — this page describes the architecture and what I learned rather than walking through solution code.

Overview

JOS is a small Unix-like kernel for x86, built in an exokernel style — most of what you'd normally think of as "kernel functionality" (fork, exec, IPC) is actually implemented in a user-level library on top of a minimal kernel, rather than inside the kernel itself. The lab sequence starts with a bare boot loader and ends with a working shell that can run programs, pipe between them, and read/write a real file system — you build every layer of that stack yourself, with the course providing only skeleton code and a grading harness.

We developed and debugged everything against QEMU (an x86 machine simulator), which made it possible to single-step through boot code, inspect page tables, and set watchpoints on kernel data structures that would be far harder to debug on real hardware.

Lab 1 — Booting a PC

The first lab is about what happens before there's an operating system at all: how a PC's BIOS hands off to a boot sector, how that boot sector (running in 16-bit real mode) switches the processor into 32-bit protected mode, and how it loads and jumps into the actual kernel from disk. We also built the kernel's early console driver and a minimal printf/backtrace implementation, then completed the lab's extra-credit challenge on top of the base exercises.

Lab 2 — Memory Management

Lab 2 builds the two foundations everything later depends on: a physical page allocator and the virtual memory system. We implemented the allocator that tracks free/used physical pages, then set up the kernel's page directory and page tables — including the "remap all of physical memory at KERNBASE" scheme JOS uses so the kernel can access any physical page through a simple offset, and the permission bits (PTE_U, PTE_W) that keep user code from touching kernel memory even though both live in the same address space. Working through this lab meant reasoning carefully about the difference between a physical address and a virtual one at every step — that distinction, and getting the page directory entries exactly right, was most of the actual difficulty.

Lab 3 — User Environments

This lab introduces environments (JOS's term for processes): the data structure representing a running program, and the machinery to create one, load an ELF binary into it, and switch the CPU into user mode to run it. Alongside that, we built the trap and exception handling path — per-vector handler stubs in assembly that normalize the stack layout before handing off to a common C dispatcher, and the interrupt descriptor table (IDT) privilege levels that decide which software interrupts user code is and isn't allowed to trigger directly (the breakpoint instruction has to be user-callable; a page fault gate has to be kernel-only, or user code could forge kernel-only exceptions). We also implemented the page-fault-driven system call path and completed this lab's extra-credit exercise.

Lab 4 — Preemptive Multitasking

Lab 4 turns JOS from a single-tasking system into a real preemptively-scheduled multiprocessor OS. This was the most involved lab: bootstrapping additional CPU cores (application processors) using a low physical bootstrap address distinct from the kernel's normal high-half linked addresses, protecting shared kernel data structures with a big kernel lock while still giving every CPU its own private kernel stack (so one CPU trapping into the kernel can never clobber another CPU's in-flight trap frame), and building a round-robin scheduler driven off the timer interrupt via sched_yield().

On top of scheduling, we implemented copy-on-write fork() — instead of copying a parent's memory into a child at fork time, both share the same physical pages marked read-only, and only get a real copy on the first write after the fork, which makes fork()+exec() (the overwhelmingly common pattern) fast since most forked pages are never actually copied. We also built JOS's inter-process communication mechanism, which deliberately skips permission checks: a message can only be delivered once the receiver has already called into sys_ipc_recv() and is explicitly waiting, so there's no way for one environment to push data into another that hasn't asked for it.

Lab 5 — File System, Spawn, and Shell

The last lab adds a real (if simple) file system, running as its own user-level environment rather than inside the kernel — in keeping with JOS's exokernel design, the file system environment is just another environment with special privilege to talk to the disk directly, communicating with everyone else over the same IPC mechanism from Lab 4. Getting that disk I/O privilege to behave correctly across context switches required understanding that x86's I/O privilege level lives in the EFLAGS register, which is already part of every environment's saved trap frame — so once the file system environment is granted I/O privilege at creation time, JOS's existing trap frame save/restore logic carries that privilege through every subsequent context switch with no extra code needed.

On top of the file system, we implemented spawn() (loading and running a new ELF binary as a child environment, JOS's equivalent of fork+exec) and got a working shell running on top of it all — able to launch programs, redirect I/O, and pipe between them, which is the point where JOS starts to feel like an actual, if minimal, Unix.

Concepts & skills demonstrated

Across the five labs, this project covers: x86 boot sequence and real-mode → protected-mode transition; physical/virtual memory management and page table permission bits; ELF loading and process/environment creation; trap, exception, and system call dispatch with IDT privilege levels; symmetric multiprocessing bootstrap and per-CPU state; locking and preemptive round-robin scheduling; copy-on-write fork; capability-free inter-process communication; and a disk-backed file system exposed through a Unix-like shell. All of it debugged directly against QEMU with GDB, working step by step from boot sector to shell.
